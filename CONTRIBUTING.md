# 기여 가이드

## 이슈 작성

- 작업 전에 이슈를 만들고 목표, 배경, 범위, 제외 범위와 완료 조건을 적어 주세요.
- 바로 구현할 기능은 `🤖 AI 개발 명세` 이슈 폼을 사용해 인수 조건과 테스트 계획을 고정합니다.
- 버그는 재현 절차, 기대 동작, 실제 동작과 실행 환경을 포함합니다.
- 결정되지 않은 정책은 구현 범위에서 제외하거나 `구현 전 확인이 필요한 결정`에 기록합니다.

## 브랜치 규칙

- `feature/이슈번호-작업명`: 새 기능
- `fix/이슈번호-작업명`: 버그 수정
- `docs/이슈번호-작업명`: 문서 변경
- `chore/이슈번호-작업명`: 설정, 의존성, 빌드 작업
- `refactor/이슈번호-작업명`: 동작 변경 없는 구조 개선
- `codex/작업명`: Codex가 수행하는 작업

한 브랜치에는 한 이슈와 한 목적만 담습니다. 작업 시작 전 `main`을 최신 상태로 맞춥니다.

## 개발 순서

1. 인수 조건을 재현하는 실패 테스트를 작성합니다.
2. 예상한 이유로 실패하는지 확인합니다.
3. 테스트를 통과시키는 최소 구현을 작성합니다.
4. 전체 테스트를 유지하며 구조를 정리합니다.
5. `./gradlew clean check`를 실행합니다.

자세한 규칙과 테스트 분류는 `docs/AI_DEVELOPMENT_HARNESS.md`를 참고합니다.

## PR 규칙

- PR 제목은 변경 내용을 짧고 명확하게 작성합니다.
- 관련 이슈를 `Closes #번호` 형식으로 연결합니다.
- 이슈의 인수 조건과 PR의 검증 결과가 서로 대응해야 합니다.
- RED-GREEN-REFACTOR 과정과 리뷰어가 집중해서 볼 부분을 기록합니다.
- 권장 변경량은 생성 파일을 제외하고 약 400줄 이하입니다. 초과하면 분리할 수 있는지 먼저 검토합니다.

## 머지 기준

- `Gradle Test`, `Docker Build`, `Gitleaks` 필수 체크가 모두 통과해야 합니다.
- 현재 단독 개발 단계에서는 작성자가 PR 체크리스트와 diff를 자체 리뷰한 뒤 머지합니다.
- 협업자가 참여하면 최소 1명 승인 규칙을 다시 활성화합니다.
- 필수 테스트나 품질 게이트를 비활성화한 PR은 머지하지 않습니다.
- 머지 후 작업 브랜치는 삭제합니다.
