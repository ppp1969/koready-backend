# KTO 실제 데이터 분석과 KoReady 활용 방안

## 0. 분석 기준

이 문서는 승인받은 한국관광공사 5개 API를 실제 호출한 결과를 바탕으로 작성한다. 단순 명세 추정이 아니라 전수 또는 명시된 확대 표본에서 계산한 값이다.

- 기준일: 2026-07-13
- 국문 관광정보 동기화 목록: 68,493건 전수
- 영문 관광정보 동기화 목록: 25,348건 전수
- 관광사진: 6,119건 전수
- 관광공모전 수상작: 96건 전수
- 연관 관광지: `baseYm=202503`, 여러 시군구 7,955건 확대 표본
- 추천 준비도 보강 분석: 국문/영문 전수 재호출 및 키워드 검색

연관 관광지는 월과 시군구를 지정해야 하므로 7,955건은 전국 전수가 아니다. 이 문서에서 연관 관광지 수치는 항상 관측 표본 수치로 해석한다.

## 1. 먼저 확정할 구현 결론

1. 국문 `areaBasedSyncList2`를 관광지 마스터의 원천으로 사용한다.
2. 전체 68,493건을 그대로 추천에 쓰지 않고 `showflag=1`을 첫 필터로 적용한다.
3. 이미지, 주소, 좌표를 모두 만족하는 카드 준비 후보는 42,517건이다.
4. 숙박과 여행코스를 제외한 온보딩 후보 기본 풀은 40,510건이다.
5. 국문과 영문 `contentid`는 직접 연결되지 않는다. 영문 `oldContentid`도 국문 ID와 일치하지 않았다.
6. 영문은 이미지 경로와 좌표+대응 관광유형으로 별도 crosswalk를 생성해야 한다.
7. 현재 규칙으로 영문 25,348건 중 20,106건(79.3%)을 단계적으로 연결할 수 있다.
8. 신분류 `lclsSystm1/2/3`를 7개 관광유형 매핑의 주 근거로 사용하고, 레거시 `cat1/2/3`는 fallback과 검증용으로 둔다.
9. 분류 코드북과 실제 데이터가 완전히 일치하지 않으므로 알 수 없는 코드를 버리거나 enum 오류로 처리하면 안 된다.
10. 연관 관광지의 ID는 TourAPI `contentid`와 다른 opaque ID다. 이름/지역/분류 기반 매칭 staging이 필요하다.
11. 연관 관광지는 `baseYm` 시점의 행정코드를 사용한다. 현재 법정동 코드로 과거 월 데이터를 조회하면 0건이 될 수 있다.
12. 관광사진의 촬영월은 추천 계절의 보조 신호일 뿐, 운영 기간이나 방문 적기의 사실값으로 사용하지 않는다.

## 5. 분류 코드 데이터 품질

### 레거시 분류 불일치

현재 코드북은 CAT1 7개, CAT2 24개, CAT3 153개지만 실제 국문 데이터에서는 각각 8개, 25개, 159개가 관측됐다.

- 데이터에만 있는 CAT1: `B01` 33건
- 데이터에만 있는 CAT2: `B0102` 33건
- 현재 CAT3 코드북에 없는 관측값: `B01020200`, `A03022600`, `B01020500`, `B01020100`, `B01020400` 등
- `cat3`에 중분류 코드인 `A0208`, `A0202`, `A0502`, `B0201`이 들어간 행도 존재
- 현재 코드북에는 있으나 전수 데이터에 없는 CAT3 코드도 존재

따라서 JPA enum으로 외부 코드를 직접 고정하지 않는다. `source_code`를 문자열로 보존하고, 내부 관광유형만 enum으로 관리한다.

### 신분류 상태

- 현재 코드북: LCLS1 10, LCLS2 59, LCLS3 245
- 실제 데이터: LCLS1 10, LCLS2 58, LCLS3 228
- LCLS1은 현재 코드북과 관측값이 일치
- LCLS2의 `VE08` 행사시설은 코드북에는 있으나 현재 전수 목록에서 관측되지 않음

신분류가 레거시보다 정합성이 좋지만 9,278건은 신분류가 비어 있다. 신분류 우선, 레거시 fallback 구조가 필요하다.

## 6. 국문/영문 처리 방식

### 전수 매칭 결과

| 매칭 단서 | 영문 행 수 |
|---|---:|
| 현재 `contentid`가 국문과 동일 | 0 |
| 영문 `oldContentid`가 국문 ID와 동일 | 0 |
| 정규화한 대표 이미지 경로 동일 | 18,209 |
| 좌표 6자리 반올림 + 대응 관광유형 동일 | 16,086 |
| 단계 규칙으로 최종 연결 | 20,106 (79.3%) |
| 최종 모호 | 1,142 |
| 최종 미연결 | 4,100 |
| 이미지와 좌표 단서 충돌 | 503 |

20,106개의 영문 행은 19,966개의 국문 관광지에 연결됐다. 국문 카드 준비 후보 중 영문 연결이 있는 장소는 13,718건이다.

### 권장 단계 매칭

1. 정규화한 이미지 경로가 국문 한 건과만 일치하면 우선 연결한다.
2. 이미지로 연결되지 않은 경우 좌표 6자리 반올림과 대응 `contenttypeid`가 한 건과 일치할 때 연결한다.
3. 후보가 여러 개면 자동 확정하지 않는다.
4. 이미지와 좌표 후보가 충돌하면 이미지 후보를 임시 우선하되 `REVIEW_REQUIRED`로 기록한다.
5. 미연결 국문 장소는 허용된 `AI_TRANSLATED` 번역으로 보강하고, 중요한 장소는 수동 교정할 수 있다.

쇼핑몰/아울렛은 여러 매장이 같은 좌표를 공유하므로 좌표만으로 연결하면 잘못된 매장에 붙을 수 있다. 제목 토큰, 주소, 대표 이미지, 관광유형을 함께 증거로 남겨야 한다.

### 필요한 DB 보강

`place_localizations.source_content_id`만으로도 최소 구현은 가능하지만, 자동 매칭 감사와 재처리를 위해 다음 테이블을 권장한다.

```text
place_source_records
- id
- provider
- language
- source_content_id
- source_old_content_id
- source_modified_time
- source_hash
- raw_snapshot_id

place_source_matches
- source_record_id
- place_id
- match_method
- confidence
- candidate_count
- evidence_json
- status: AUTO_CONFIRMED / REVIEW_REQUIRED / REJECTED
- matcher_version
```

## 8. 연관 관광지 활용

다지역 표본 7,955건에서 다음이 관측됐다.

- 기준 장소: 243개
- 연관 장소: 2,609개
- 연관 쌍: 7,955개
- 순위: 1~50
- 대분류: 관광지 3,882, 음식 3,340, 숙박 733
- 중분류: 9개
- 소분류: 42개
- 주요 소분류: 한식 1,200, 카페/찻집 1,030, 전문음식 563, 자연경관 526, 호텔 489, 전시시설 381, 시장 369

`tAtsCd`와 `rlteTatsCd`는 32자리 opaque ID이며 국문 TourAPI `contentid`와 형태가 다르다. `place_relations`에 바로 FK로 넣지 말고 다음 staging을 거친다.

```text
related_place_staging
- base_ym
- source_tats_code / source_name / source_region
- related_tats_code / related_name / related_region
- category_l1/l2/l3
- related_rank
- matched_source_place_id
- matched_related_place_id
- match_status
- match_evidence_json
```

화면에는 `match_status=AUTO_CONFIRMED|MANUAL_CONFIRMED`인 관계만 사용한다. 매칭되지 않은 행도 버리지 않고 다음 동기화/수동 검수 대상으로 남긴다.

### 행정코드 시점 문제

현재 법정동 코드 API는 16개 시도와 `전남광주통합특별시(12)`를 반환하지만, 연관 관광지 표본의 기준월은 2025-03이다. 과거 데이터는 당시 행정코드를 요구할 수 있다. 현재 코드북을 과거 월 조회에 그대로 사용하면 정상 응답이면서 0건이 나올 수 있다.

`administrative_regions`에 다음을 추가한다.

- `code_system`: TOUR_AREA / LEGAL_DONG / RELATED_TOUR
- `effective_from`, `effective_to`
- `source_base_ym`
- `replaced_by_code`

## 9. 기능별 실제 활용

### 온보딩 후보 10개

- 원천 풀: `onboardingClassified=40,510`
- 사용자별 동적 추출이 아니라 관리자가 정확히 10개를 검수해 버전형 세트로 발행
- 발행 검수 권장 기준은 같은 `travelStyle` 최대 2개, 같은 7권역 최대 3개
- 음식/쇼핑 과대표집을 피하도록 관리자 화면에 유형·권역 분포 경고 표시
- DRAMA_LOCATION은 운영자 seed와 AI 검수 후보를 별도 유지
- 이미지, 제목, 주소, 좌표, 영문 또는 번역 설명이 있는 후보만 발행 가능
- 선택 관광지의 `place_tags`를 `user_preference_tags`로 복사하되 source place를 보존

### 월별 추천

- 축제는 `detailIntro2.eventstartdate/eventenddate`와 `searchFestival2`를 사실값으로 사용
- 현재 표출 가능한 축제는 922건, 카드 준비 축제는 912건
- 축제 노출 기간은 `[시작일 - 6개월, 종료일]`로 고정하며 범위 밖의 축제는 추천하지 않음
- 일반 명소는 사진 촬영월, 수상작 계절 키워드, AI 계절 태그를 보조 점수로 사용
- 촬영월 하나만으로 추천월을 확정하지 않음
- `reason_type=EVENT_PERIOD`를 `PHOTO_SEASON_HINT`보다 우선

### 지도 추천

- 표출 가능하며 좌표가 있는 50,622건이 최대 지도 후보
- 프론트가 제공하는 7개 권역 중 사용자가 클릭한 `serviceRegionCode`로 장소를 필터링한다.
- 지도 도형과 권역 버튼은 프론트 정적 asset이며 서버는 `bbox`, `zoom`, GPS 좌표를 받지 않는다.
- 초기 화면에서 전부 반환하지 않고 권역 필터와 cursor pagination으로 목록을 제공한다.
- 응답 DTO에는 좌표, 내부 `placeId`, 대표 유형, 이미지 여부를 우선 반환한다.

### K-Local Pick

- 카드 준비 42,517건에서 숙박/여행코스 정책을 적용
- 관광유형과 취향 태그 bucket을 만든 뒤 품질 점수와 권역 다양성을 적용
- `showflag=0`, 이미지 없음, 좌표 없음은 기본 제외
- 영문 사용자는 영문 crosswalk 또는 번역이 준비된 후보를 우선
- 추천 순서는 서버 deck에 고정하고 `CARD_SERVED` 직후 노출 상태를 기록

### 상세 페이지

- `detailCommon2`: 공통 설명, 홈페이지, 전화, 주소
- `detailIntro2`: 콘텐츠 유형별 운영/요금/주차/기간
- `detailInfo2`: 코스, 객실, 반복 상세
- `detailImage2`: 추가 이미지
- 외부 필드명을 프론트 DTO에 노출하지 않고 내부 표준 필드로 정규화

### Buddy Route/Hori Tip

- Route 출발점은 사용자 저장 위치, 도착점은 국문 관광지 좌표
- 좌표가 없는 표출 장소는 Route 버튼 비활성 또는 제외
- 운영시간, 휴무일, 주차, 반려동물 필드는 Hori Tip의 장소 정보 보조 신호
- 위 신호로 팁 본문을 자동 생성하지 않는다. 운영진이 Hori Tip을 작성하고 Route trigger와 대상 장소를 지정한다.
- 실제 이동시간/환승/당일치기는 TourAPI가 아니라 TMAP 응답으로 판단

## 10. 수집과 동기화 권장 순서

```text
1. 지역/분류 코드북 snapshot 저장
2. 국문 areaBasedSyncList2 전수 또는 증분 수집
3. showflag 포함 places upsert
4. 신규/수정 장소 상세 작업 queue 생성
5. detailCommon/Intro/Info/Image 보강
6. 영문 sync 수집 및 place_source_matches 생성
7. 관광사진/수상작 이미지 후보 매칭
8. 연관 관광지 staging 및 매칭
9. 관광유형/태그 규칙 실행
10. 추천 품질 점수와 월별 feature 계산
```

실험에서는 `numOfRows=70000`으로 국문/영문 전수를 각각 한 번에 받을 수 있었다. 운영 배치는 큰 응답 한 번보다 500~1,000건 페이지를 권장한다. 실패 재시도 범위가 작고 cursor/checkpoint 관리가 쉽기 때문이다.

### 동기화 검증 규칙

- 페이지 합계가 `totalCount`와 다르면 batch 실패
- 한 페이지가 중간에 비면 batch 실패
- `contentid` 중복 감지
- 알 수 없는 분류 코드는 경고 후 원문 저장
- `showflag` 변화는 hard delete가 아니라 active 전환
- 수정시간 역행, 행정코드 변경, 이미지 URL 변경을 별도 지표로 기록
- API 키와 토큰은 로그/원문 snapshot에 저장하지 않음

## 11. 구현 우선순위

### 1차

1. `places`, `place_source_records`, 코드북 snapshot
2. 국문 sync batch와 호출 로그
3. 상세 보강 queue
4. `place_style_mappings` 규칙 v1
5. 온보딩 40,510 후보에서 quota 기반 10개 생성
6. 추천 deck/cursor/no-repeat

### 2차

1. 영문 crosswalk와 번역 fallback
2. 관광사진/수상작 이미지 매칭
3. 월별 feature 생성
4. 연관 관광지 staging과 검수

### 구현 완료 검증 기준

- 국문 동기화 입력 수와 upsert+비표출 처리 수 합계가 일치
- 모든 외부 ID/코드는 문자열 원문 보존
- 온보딩 후보가 7개 유형/7개 권역 quota를 만족
- 같은 추천 deck에서 장소 중복 0
- 동일 cursor 재요청 결과가 동일
- 영문 자동 매칭 20,106건에 matcher version과 evidence 저장
- 모호 1,142건과 미연결 4,100건이 자동 확정되지 않음
- `showflag=0` 장소가 사용자 추천에 노출되지 않음
