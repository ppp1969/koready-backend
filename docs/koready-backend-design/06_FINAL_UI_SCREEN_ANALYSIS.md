## 1. 최종 화면 범위

| 영역 | 확인 화면 | 백엔드 범위 |
|---|---|---|
| 로그인 | 스플래시, Google/Apple 로그인 | 인증/토큰 |
| 약관 | 필수 약관, 마케팅 선택 동의 | 약관 버전/동의 이력 |
| 언어 | 한국어, English | 선호 언어 |
| 온보딩 | 위치, 여행 스타일, 선호 여행지 | 사용자 취향/위치/후보 |
| 메인 | 기본 주소, 언어, 월별 추천 | 홈/월별 추천 |
| 추천 | 카드 앞/뒤, 스와이프, 저장, 경로 진입 | 추천 덱/이벤트/저장 |
| 장소 | 설명, 이동, 메이트 탭 | 장소 상세/경로/메이트 |
| 경로 | 요약, 상세 구간, 비용, Hori Tip | TMAP 가공 응답 |
| 메이트 | 프로필 설정/목록/상세 | Buddy 프로필 |
| 쪽지 | 작성, 쪽지함, 스레드, 답변 | 1,000자 쪽지 |
| 한국 여행 가이드 | KTX 예매 가이드·시뮬레이션 | 프론트 정적 콘텐츠, 백엔드 API 없음 |

하단 내비게이션에는 홈, 지도, 추천, 저장, 마이가 보이지만 지도·저장·마이의 독립 최신 섹션은 현재 페이지에서 확인되지 않는다. 지도는 기존 7개 권역 선택 구조를 사용한다. 권역 도형과 버튼은 프론트 정적으로 제공하고, 백엔드는 선택한 `serviceRegionCode`의 장소 목록만 cursor 방식으로 반환한다. GPS, `bbox`, `zoom` 기반 조회는 구현하지 않는다.

## 2. 로그인과 온보딩

## 2.1 실제 흐름

```text
스플래시
  -> Google/Apple 로그인
  -> 약관 동의
  -> KO/EN 언어 선택
  -> 현재 위치 검색/선택
  -> 여행 스타일 1~4개 선택
  -> 선호 여행지 1~3개 선택
  -> 완료
```

최종 온보딩은 위치 `01`, 여행 스타일 `02`, 선호 여행지 `03`의 세 단계다. 화면 번호 자체를 서버 상태 코드로 사용하지 않고 `LOCATION`, `TRAVEL_STYLES`, `PREFERENCE_PLACES`, `COMPLETED`만 사용한다.

## 2.2 값

### 삭제 확정된 단계

이전 Figma의 한국 방문 목적 화면은 구현하지 않는다. 관련 선택값, API 필드, enum, DB 컬럼도 만들지 않는다.

### 여행 스타일

- 로컬 맛집
- 지역 축제
- 전통시장
- 문화체험
- 자연 명소
- 전시/미술관
- 드라마 촬영지

중복 없이 1~4개를 선택한다. `숨은 카페`는 최종 온보딩 값이 아니다.

### 선호 여행지

- 관리자가 검수해 발행한 후보 정확히 10개
- 1~3개 선택
- 후보 예시의 장소명과 태그는 enum이 아니며 관리자 큐레이션 데이터다.
- 발행본은 버전형으로 관리하고 프론트는 조회한 `candidateSetId/version`을 완료 요청에 포함한다.

### 위치

- 주소/학교/동네 검색 결과만 사용한다.
- GPS는 사용하지 않는다.
- Figma의 `현재 위치로 찾기` 버튼은 삭제 대상이다.

## 3. 메인과 월별 추천

### 화면 데이터

- 기본 위치 표시/변경
- 한국어/English 토글
- 현재 월 추천 미리보기
- 전체보기의 1~12월 선택
- 추천순/마감순
- 지역, 날짜, 관광유형 필터
- KTX 여행 가이드 진입

### 백엔드 판단

- 홈 API는 위치, 언어, 월별 추천 미리보기만 제공한다.
- KTX 가이드 카드/본문/이미지는 프론트 번들에 포함한다.
- 월별 추천 날짜는 `Asia/Seoul` 기준으로 계산한다.
- 월별 전체보기는 개최 `year`와 `month`를 함께 사용해 반복 축제의 연도별 회차를 구분한다.
- 축제는 시작 6개월 전부터 상세 목록에 보이며, 종료 뒤에도 해당 개최 연도·월에는 `종료` 상태로 남는다.
- 홈과 현재 월 추천은 진행 중·예정 축제를 종료 축제보다 우선한다.
- 지역 필터는 확정된 7개 권역을 사용한다. Figma의 `강원` 중복 표기는 화면 오타다.

### KTX 가이드 역할 분담

- 기획은 가이드별 화면 순서, 단계별 제목·본문·버튼 문구·주의사항, 사용할 이미지 예시를 정리해 프론트에 전달한다.
- 프론트는 코레일톡 화면을 따라가는 가이드와 시뮬레이션을 정적 데이터와 asset으로 구현한다.
- 백엔드는 KTX 가이드용 API, DB, 관리자 CMS를 MVP에 만들지 않는다.
- 영상 가이드와 오디오 가이드는 MVP에서 제외한다.

## 4. K-Local Pick

### 화면 데이터

- 범위: 근교/전국
- 카드 앞면: 이미지, 제목, 지역, 저장 상태
- 카드 뒷면: 태그, 추천 설명, 여행 코스 확인
- 이전/다음 스와이프

### 백엔드 판단

- `NEARBY`는 기본 위치의 서비스 권역 기준이며 GPS 거리가 아니다.
- 같은 추천 덱에서 장소가 중복되면 안 된다.
- 서버가 응답에 포함한 `CARD_SERVED` 장소는 30일 동안 새 덱에서 제외한다.
- 스와이프는 좋아요/싫어요가 아니라 탐색 이벤트다.

## 6. Buddy Route

### 요약 화면

- 출발지/목적지
- 예상 편도 소요시간
- 추천 교통수단
- 이동 난이도
- 당일치기/숙박 권장
- 예상 편도/왕복 요금
- 상세 경로 진입

### 상세 화면

- 구간별 출발/도착지
- 교통수단/노선
- 구간별 소요시간
- 운행 여부와 안내 문구
- Hori Tip
- 선택적 예매/시간표 외부 링크

여기서 Hori Tip은 운영진이 백엔드에 저장한 짧은 경로 안내다. KTX 예매 화면을 단계별로 따라가는 정적 가이드와는 별도 기능이다.

### 확정 보정

- 현재 Figma의 `약 3시간 10분 + 당일치기 가능`은 잘못된 샘플이다.
- 편도 180분 이상이므로 해당 화면은 `숙박 권장`으로 수정해야 한다.
- `당일치기는 가능하지만`이라는 Hori Tip 문구도 숙박 권장 문구로 수정해야 한다.

### 데이터 원천

- TMAP: 총 시간, 총요금, 환승, 도보, 구간, 교통수단, 노선, 출발/도착 정보
- Koready: 난이도, 당일치기 여부, Hori Tip, 표시용 교통수단 조합
- 별도 가공: 축제 셔틀버스와 시간표 링크

TMAP 상세 결과는 24시간 이상 저장/사용하지 않는다.

## 7. Buddy Connect와 쪽지

### Buddy 프로필

- 프로필 이미지
- 닉네임
- 국적
- 사용 언어
- 한국어 수준: 초급/중급/고급
- 한 줄 소개: 최대 500자
- 관심 스타일
- 프로필/SNS/쪽지 공개 설정

Buddy 프로필 관심 스타일은 온보딩 여행 스타일과 다른 카탈로그다.

### 쪽지

- 실시간 채팅이 아닌 스레드형 쪽지
- 첫 쪽지와 답변 모두 1~1,000자
- 현재 Figma의 `0/500` 카운터는 `0/1000`으로 수정해야 한다.
- 차단, 신고, 쪽지 수신 거부가 필요하다.
