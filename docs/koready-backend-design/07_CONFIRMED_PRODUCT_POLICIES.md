# Koready 확정 제품 정책

## 0. 문서 우선순위

이 문서는 2026-07-17 PM 검토까지 반영한 개발 전 제품 정책이다. 기존 문서의 `초안`, `팀 결정 필요`, `운영 설정으로 변경 가능` 문구와 충돌하면 이 문서를 우선한다.

## 1. 핵심 타깃과 온보딩

- 핵심 타깃은 외국인 유학생, 교환학생, 장기체류 외국인이다.
- 단기 여행자는 핵심 타깃 검증 이후 확장할 사용자군이다.
- 온보딩은 `위치 입력 -> 여행 스타일 1~4개 -> 관심 관광지 1~3개` 순서다.
- 위치는 사용자가 Kakao 검색 결과를 선택해 먼저 저장한다.
- 여행 스타일은 중복 없이 1~4개를 선택한다.
- 관심 관광지는 관리자가 발행한 10개 후보 중 1~3개를 선택한다.
- 방문 목적 단계는 사용하지 않으며 관련 API 필드, enum, DB 컬럼을 만들지 않는다.
- 서버 온보딩 단계는 `LOCATION`, `TRAVEL_STYLES`, `PREFERENCE_PLACES`, `COMPLETED`다.

## 2. 서비스 권역

서비스 권역은 7개다.

```text
SEOUL
GYEONGGI      = 경기도 + 인천광역시
GANGWON
CHUNGCHEONG   = 대전 + 세종 + 충청북도 + 충청남도
JEOLLA        = 광주 + 전북 + 전남
GYEONGSANG    = 부산 + 대구 + 울산 + 경북 + 경남
JEJU
```

외부 API 지역코드는 원문으로 보존하고 `service_region_code`로 별도 매핑한다.

## 3. 위치 저장/삭제

- 사용자는 배달앱처럼 주소 검색 결과를 여러 개 저장할 수 있다.
- 위치 검색은 Koready 백엔드가 Kakao Local API의 주소 검색과 키워드 장소 검색을 호출해 정규화한다.
- 프론트는 Kakao REST API key를 보유하거나 Kakao Local API를 직접 호출하지 않는다.
- 검색 응답에는 원본 필드와 함께 서버가 서명한 단기 `searchResultToken`을 포함한다.
- 위치 저장 요청은 주소·좌표를 다시 제출하지 않고 `searchResultToken`, 사용자 별칭, 기본 위치 지정 여부만 제출한다.
- 서버는 token 서명과 만료를 검증한 뒤 검색 당시의 주소·좌표·권역을 저장한다.
- 저장 위치 중 삭제되지 않은 한 개를 기본 위치로 지정할 수 있다.
- 위치 추가, 목록 조회, 기본 위치 변경, 개별 삭제를 지원한다.
- 삭제는 `deleted_at` 기반 soft delete이며 사용자 화면과 추천/경로에서는 즉시 사라진다.
- 기본 위치 삭제 시 남은 위치가 있으면 새 기본 위치를 지정한다.
- 저장 위치가 하나도 없으면 추천과 Buddy Route 진입 전에 위치 등록을 요구한다.
- 삭제 위치의 물리 보존/파기 기간은 개인정보처리방침에서 별도 확정한다.

### GPS 정책

- GPS와 기기 현재 위치를 사용하지 않는다.
- 사용자가 주소, 학교, 동네를 검색하고 선택한 결과만 위치로 저장한다.
- 위치 검색 화면의 `현재 위치로 찾기`는 구현 대상이 아니다.

### 검색 개인정보와 장애 정책

- 검색어, 상세 주소, Kakao 원본 응답은 장기 호출 로그와 공모전 증빙에 포함하지 않는다.
- 동일 검색어의 짧은 캐시는 허용하되 사용자별 검색 이력을 목적으로 보존하지 않는다.
- 검색 결과 없음은 성공 응답의 빈 `items`로 반환한다.
- Kakao 호출 실패와 timeout은 `503 LOCATION_PROVIDER_UNAVAILABLE`, KoReady 호출 제한은 `429 RATE_LIMITED`로 구분한다.

## 4. 온보딩 선호 여행지 큐레이션

- 온보딩 마지막 단계의 선호 여행지 후보는 사용자별 동적 추천으로 생성하지 않는다.
- 관리자가 KTO 기반 `places` 중 정확히 10개를 골라 하나의 큐레이션 세트로 발행한다.
- 프론트는 현재 발행 세트를 조회하고 사용자는 그중 1~3개를 선택한다.
- 큐레이션 세트 상태는 `DRAFT`, `PUBLISHED`, `ARCHIVED`만 사용한다.
- 초안은 자유롭게 수정할 수 있지만 발행된 버전은 수정하지 않는다. 변경 시 새 버전을 발행한다.
- 사용자가 세트를 조회한 뒤 새 버전이 발행되더라도, 조회했던 발행 버전으로 온보딩을 완료할 수 있다.
- 발행 조건은 후보 10개, 중복 장소 없음, 활성/표출 가능 장소, 한국어 제목·주소·좌표·대표 이미지 보유다.
- 항목별 표시 순서, 대표 이미지 선택, KO/EN 큐레이터 한 줄 설명과 노출 태그를 관리자가 편집할 수 있다.
- 발행·보관과 후보 변경은 관리자 감사 로그에 남긴다.

## 5. 추천 재노출

- 같은 deck 안의 `place_id` 중복은 허용하지 않는다.
- 사용자에게 `CARD_SERVED`된 장소는 30일 동안 새 추천 deck에 포함하지 않는다.
- `suppress_until = served_at + 30일`이다.
- 동일 cursor 재요청은 동일 결과를 반환하고 suppress 기간을 중복 연장하지 않는다.
- 후보가 부족해도 30일을 줄이지 않는다.
- 후보가 없으면 3순위와 전국 범위까지 확장하고, 그래도 없으면 `hasMore=false`를 반환한다.
- MVP v1은 기록 시점, 제한 일수, 정책 버전을 각각 `served_at`, `30`, `recommendation-suppression-v1`로 함께 보존한다.
- 실제 서비스에서 후보 소진율과 빈 덱 비율을 측정해 후보가 지나치게 빨리 줄어드는 근거가 확인되면 새 정책 버전에서 기록 시점 또는 기간을 조정한다.
- 새 정책은 기존 노출 이력과 `suppress_until`을 소급 변경하지 않는다.

## 6. AI 번역

- 영문 TourAPI 연결을 먼저 시도한다.
- 영문 원천이 없는 장소는 국문 원천의 AI 번역을 허용한다.
- `translation_source=AI_TRANSLATED`를 저장한다.
- 원문 hash, 모델명, prompt version, 생성 시각, 검수 상태를 저장한다.
- 국문 원천이 변경되면 기존 번역을 stale로 표시하고 재생성한다.
- 운영자 수정본은 `MANUAL_EDITED`로 승격하며 자동 재생성으로 덮어쓰지 않는다.

## 7. 월별 축제 회차와 노출

기준 타임존은 `Asia/Seoul`, 비교 단위는 `LocalDate`다.

```text
visibleFrom = eventStartDate.minusMonths(6)
```

- 월별 상세 목록은 행사 개최 `year`와 `month`를 함께 받아 연도별 회차를 구분한다.
- 반복 축제는 축제 시리즈와 개최 회차를 분리하고, 각 회차에 `eventYear`, `startDate`, `endDate`를 저장한다.
- 시작일까지 6개월을 초과해 남은 회차는 아직 상세 목록에 노출하지 않는다.
- `visibleFrom` 이후에는 종료된 회차도 해당 개최 연도·월 상세 목록에 남긴다.
- 상태는 저장하지 않고 조회 시점에 `UPCOMING`, `ONGOING`, `ENDED`로 계산한다.
- 홈과 현재 월 추천은 `ONGOING`, `UPCOMING`을 `ENDED`보다 우선한다.
- 시작일/종료일이 없거나 파싱할 수 없으면 날짜 기반 추천에서 제외한다.
- 시작일 6개월 전과 종료일 당일은 각각 노출 시작 경계와 `ONGOING` 경계에 포함한다.
- 이전 연도 회차의 기간을 신규 연도 회차에 덮어쓰거나 섞지 않는다.

## 8. Buddy Route

- 대중교통 경로는 TMAP 대중교통 API가 제공하는 범위에서 최신 UI 구조로 제공한다.
- TMAP 결과는 Koready의 요약/구간 DTO로 정규화하고 원문을 프론트에 노출하지 않는다.
- 편도 소요시간이 180분 미만이면 `DAY_TRIP_AVAILABLE`이다.
- 편도 소요시간이 180분 이상이면 `STAY_RECOMMENDED`다.
- 난이도와 당일치기 여부는 Koready 백엔드가 계산한다.
- 사용자에게 표시하는 이동 안내 명칭은 `Hori Tip`이다.
- Hori Tip 본문은 운영진이 백엔드에 저장하고, Route 조회 시점에 목적지·구간·노출 기간과 매칭해 응답에 조합한다.
- Hori Tip은 이동 경로에서 노출되는 짧은 안내이며 KTX 예매 가이드·시뮬레이션과 별도 기능이다.
- TMAP 원본과 시간·요금·구간 등 정규화 결과는 24시간 미만 캐시에만 두고 만료 시 삭제한다.
- 장기 DB에는 요청 시각, 성공 여부, 오류 코드, 지연시간처럼 경로 결과를 재현하지 않는 호출 메타데이터만 저장한다.
- 축제 셔틀버스처럼 TMAP에 없는 정보는 별도 가공 데이터가 있을 때만 보강한다.

## 9. Buddy 프로필과 쪽지

- 한국어 수준은 `BEGINNER`, `INTERMEDIATE`, `ADVANCED` 세 값만 사용한다.
- 쪽지 본문은 공백 제거 후 1~1,000자다.
- 프로필 한 줄 소개는 쪽지와 별도 필드이며 최대 500자를 유지한다.
- 실시간 채팅이 아니라 스레드형 쪽지로 제공한다.

## 10. 지도와 지역별 장소 조회

- 지도 도형, 지역명, 7개 `ServiceRegionCode` 버튼은 프론트 정적 asset으로 제공한다.
- 사용자가 지역을 클릭하면 백엔드에 `serviceRegionCode`를 전달해 해당 지역의 장소 목록을 조회한다.
- 지도 API는 GPS, 사용자 실시간 좌표, viewport `bbox`, `zoom`을 사용하지 않는다.
- 장소 목록은 cursor pagination을 사용하고 관광유형·추천순/마감순 필터를 선택적으로 적용한다.

## 11. KTX 가이드

- 코레일톡 화면을 따라가는 KTX 예매 가이드·시뮬레이션은 프론트 정적 콘텐츠로 제공한다.
- MVP 백엔드 API, DB, 배치 대상에서 제외한다.
- 운영 중 콘텐츠 업데이트 기능은 현재 만들지 않는다.
- 기획은 화면 순서, 단계별 제목·본문·버튼·주의 문구, 이미지 예시와 출처를 정리해 프론트에 전달한다.
- 프론트는 전달받은 문구와 이미지를 정적 데이터·asset으로 구성하고 이전/다음/완료 동작을 구현한다.
- 영상 가이드와 오디오 가이드는 MVP 범위에서 제외한다.

## 12. OpenAPI 공모전 증빙

- 모든 외부 API 호출은 성공/실패와 관계없이 마스킹된 호출 로그를 남긴다.
- 성공한 KTO 호출은 가공 전에 원천 응답 snapshot을 private object storage에 immutable 파일로 저장한다.
- snapshot에는 call log ID, SHA-256, byte 크기, 수집 시각과 보관 등급을 연결한다.
- KTO 원천 snapshot과 호출 로그는 공모전 증빙 번들에 포함할 수 있어야 한다.
- TMAP 원천/정규화 경로 데이터는 provider 약관에 따라 24시간 전에 삭제하고 장기 증빙 번들에서 제외한다.
- 관리자 snapshot/증빙 다운로드, 배치 수동 실행, cursor 초기화는 감사 로그를 남긴다.
- `serviceKey`, `appKey`, token 등 secret은 로그/snapshot manifest/API 응답에 노출하지 않는다.
- TMAP/KAKAO의 사용자 좌표, 상세 주소, 검색어는 장기 호출 로그와 증빙 번들에 포함하지 않는다.

## 13. 구현 구조 확정

- refresh token은 `auth_sessions`에 원문이 아닌 hash로 저장하고 회전·로그아웃·폐기 이력을 관리한다.
- 기본 위치의 단일 원천은 `users.default_location_id`다.
- 온보딩 선호 여행지의 단일 원천은 관리자가 발행한 불변 버전의 큐레이션 세트다.
- 위치 검색 결과 선택은 서버가 발행한 단기 `searchResultToken`으로 검증한다.
- `CARD_SERVED`는 추천 응답 시 서버만 기록하며 클라이언트 이벤트 enum에 포함하지 않는다.
- 축제 master와 연도별 개최 회차는 분리하며 `(festival_series_id, event_year, occurrence_sequence)`를 회차 유일성 기준으로 사용한다.
- Buddy 관심 스타일은 온보딩 여행 스타일과 분리된 여섯 개 `BuddyStyle`을 사용한다.
- 쪽지는 `buddy_message_threads`와 `buddy_messages.thread_id`로 구성하며 `Idempotency-Key`는 DB unique 제약으로 보호한다.
- 당일치기 판정은 TMAP 원본 초 값으로 수행하고 표시용 분 반올림과 분리한다.
- 외부 API provider는 `KTO/TMAP/KAKAO/AI`, 소셜 provider는 `GOOGLE/APPLE`로 분리한다.
- 번역 출처는 `KTO_KO/KTO_EN/AI_TRANSLATED/MANUAL_EDITED`만 사용한다.
- batch status는 `PENDING/RUNNING/COMPLETED/FAILED/PARTIAL_FAILED`로 통일한다.
- 원천 snapshot에는 원문 SHA-256과 실제 gzip 객체 SHA-256을 각각 저장한다.
- 월별 추천에서 월과 날짜 필터를 함께 보내면 선택 월 범위 안에서 날짜 필터를 교집합 적용한다.
- 장소 상세의 운영시간·운영기간·휴무·요금·주차는 nullable 표준 필드로 같은 응답에 포함한다.
- OpenAPI 문서를 endpoint와 schema의 기계 판독 가능한 최종 기준으로 사용하고 Markdown은 정책·설명 자료로 사용한다.

## 14. 개발 수락 기준

- 인천 주소가 `GYEONGGI`로 매핑되는 테스트가 통과한다.
- 사용자별 활성 기본 위치가 둘 이상 생기지 않는다.
- 삭제 위치는 위치 목록, 추천, 경로에서 조회되지 않는다.
- 온보딩 요청과 응답, enum, DB에 방문 목적 항목이 존재하지 않는다.
- 온보딩 단계가 위치, 여행 스타일, 관심 관광지 순서로 복구된다.
- `CARD_SERVED` 후 29일에는 미노출, 30일 경과 후에는 후보 복귀가 가능하다.
- 동일 cursor 재시도로 suppress 기간이 연장되지 않는다.
- 영문 원천 없음 시 AI 번역이 생성되고 출처/버전이 저장된다.
- 시작 6개월 전부터 축제 회차가 보이고 종료 뒤에도 해당 개최 연도·월 목록에는 `ENDED`로 남는다.
- 같은 축제의 서로 다른 개최 연도 회차가 별도 ID와 기간을 가지며 서로 덮어쓰지 않는다.
- 홈과 현재 월 추천에서 진행 중·예정 회차가 종료 회차보다 먼저 정렬된다.
- GPS 또는 기기 현재 위치 기반 endpoint가 존재하지 않는다.
- 프론트 요청만으로 위치 주소·좌표·권역을 위변조할 수 없다.
- 발행 후보가 정확히 10개가 아니거나 중복/비표출 장소가 있으면 큐레이션 세트 발행이 거절된다.
- 발행 후 새 버전이 생겨도 사용자가 조회한 이전 발행 버전의 유효한 1~3개 선택을 저장할 수 있다.
- 지역별 장소 조회는 `serviceRegionCode`만으로 동작하며 `bbox/zoom`을 요구하지 않는다.
- 편도 179분은 당일치기 가능, 180분은 숙박 권장으로 판정한다.
- 쪽지 1,000자는 허용하고 1,001자는 거절한다.
- 한국어 수준 `NONE`, `NATIVE`는 거절한다.
- 이동 안내의 사용자 노출 제목은 `Hori Tip`이다.
- KTX 가이드 본문은 홈 API 응답에 포함하지 않는다.
- 영상·오디오 가이드용 API가 존재하지 않는다.
- 성공한 KTO 호출의 call log, raw snapshot, SHA-256 계보를 조회할 수 있다.
- 일반 사용자 token으로 관리자 API를 호출하면 403을 반환한다.
- TMAP 원천 데이터는 공모전 장기 증빙 파일에 포함되지 않는다.
