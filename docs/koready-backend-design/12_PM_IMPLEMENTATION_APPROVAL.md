> PM이 개발 용어를 몰라도 제품 범위와 정책을 판단할 수 있도록 다시 작성한 설명본
>
> - 문서 상태: 2026-07-17 PM 1차 검토 반영, 최종 개발 착수 승인 요청
> - 기준일: 2026-07-17 (Asia/Seoul)
> - 제품 기준: KOREADY Figma UI와 2026-07-17까지 확정된 정책
> - API 기준: 현재 Swagger(API 사용 설명서)에 공개된 기능 명세
> - 데이터 근거: 한국관광공사(KTO) 실제 데이터 조사와 TMAP 시험 호출
> - 승인 대상: 사용자에게 제공할 기능, 운영 규칙, 데이터 처리 방식, 개발 순서, MVP 범위

## 0. 이 문서를 읽는 방법

이 문서는 코드를 승인받기 위한 문서가 아니다. **Koready가 어떤 서비스로 동작해야 하는지, 어떤 규칙으로 데이터를 보여줄지, 지금 가진 데이터로 실제 구현이 가능한지**를 PM에게 확인받기 위한 문서다.

PM은 다음 네 가지만 중점적으로 보면 된다.

1. 사용자가 앱에서 겪는 순서와 화면 동작이 기획 의도와 같은가.
2. 추천, 축제, 위치, 경로, Hori Tip, Buddy Connect의 운영 규칙이 맞는가.
3. MVP에 넣을 기능과 뒤로 미룰 기능이 맞는가.
4. 문서 마지막의 **PM 승인 요청표**에 승인, 수정, 보류를 표시할 수 있는가.

개발자가 참고할 숫자, API 주소, 시스템 내부 코드명은 뒤쪽과 부록에 남겼다. PM이 그 값을 외울 필요는 없다.

### 문서에서 사용하는 표시

| 표시 | 뜻 |
|---|---|
| **확정** | 이전 논의에서 이미 결정되어 그대로 개발하려는 내용 |
| **승인 요청** | 개발 전에 PM의 승인 또는 수정이 필요한 내용 |
| **실제 데이터 확인** | 외부 서비스를 직접 호출해 데이터가 오는지 확인한 내용 |
| **조건부 가능** | 구현은 가능하지만 운영 검수, 번역, 저작권 확인 등이 더 필요한 내용 |
| **아직 확인 전** | 설계만 있고 외부 서비스와 실제 연결은 아직 하지 않은 내용 |

## 1. 먼저 보는 결론

### 1.1 지금 어디까지 준비됐나

| 구분 | 현재 상태 | 쉽게 말하면 |
|---|---|---|
| 테스트 서버 | Render에 준비됨 | 인터넷에서 서버와 Swagger 문서를 열어볼 수 있음 |
| 테스트 DB | Aiven MySQL 연결 완료 | 앞으로 개발할 데이터를 저장할 공간은 준비됨 |
| 자동 검사 | 준비됨 | 코드가 올라오면 테스트, Docker 빌드, 비밀정보 포함 여부를 자동 확인함 |
| 화면 분석 | 완료 | 현재 Figma 화면에서 서버가 제공해야 할 데이터를 정리함 |
| API 설계 | 71개 기능 명세 작성 | 프론트와 백엔드가 주고받을 데이터의 약속을 먼저 작성함 |
| 실제 비즈니스 기능 | 아직 시작 전 | Swagger에 보인다고 실제 추천·저장·경로 기능이 동작하는 것은 아님 |
| KTO 데이터 조사 | 1차 완료 | 장소·축제·사진 데이터를 실제로 받아 구현 가능 범위를 확인함 |
| TMAP 조사 | 제한적으로 완료 | 경로 3건을 시험해 기본 응답과 오류 모양을 확인함 |
| Kakao·로그인·AI | 실제 연동 전 | 해당 기능 개발 직전에 소규모 시험이 더 필요함 |

현재 Swagger의 71개 기능은 모두 **PLANNED**, 즉 “설계만 작성됨” 상태다. 실제 기능 코드와 서비스용 DB 구조는 아직 없다.

### 1.2 이번 승인으로 시작되는 일

이 문서가 승인되면 다음 순서로 시작한다.

1. 장소, 축제, 추천에 필요한 DB 구조를 만든다.
2. 한국관광공사 데이터를 안전하게 수집하고 정리한다.
3. 장소 조회, 월별 축제, 온보딩 후보 10개, K-Local Pick 추천을 개발한다.
4. 로그인과 위치 저장은 위 기능이 어느 정도 준비된 뒤 연결한다.
5. TMAP 경로와 Buddy 기능은 그다음 단계에서 개발한다.

첫 DB 구조는 한번 테스트 서버에 적용한 뒤 함부로 고치기 어렵다. 그래서 **어떤 데이터를 중복 없이 저장할지와 어떤 제품 정책을 적용할지 먼저 승인**받는다.

### 1.3 PM이 최종적으로 답할 질문

- 핵심 사용자를 외국인 유학생·교환학생·장기체류 외국인으로 잡는 것이 맞는가.
- 온보딩, 추천, 축제, 경로, Buddy 기능의 사용자 흐름이 맞는가.
- Hori Tip과 KTX 가이드의 프론트·백엔드 역할 분담이 맞는가.
- KTO 기능을 먼저 개발하고 로그인과 TMAP을 뒤로 미루는 순서가 맞는가.
- 아직 확인이 필요한 위험 요소를 감수하고 1단계 개발을 시작해도 되는가.

| 캐시 | 짧은 시간 동안 결과를 임시 보관해 다시 계산하지 않는 공간 | TMAP 경로를 약 30분 보관 |

| PR | 한 가지 개발 변경을 검토하고 합치는 작업 단위 | 장소 DB 구조만 담은 첫 번째 개발 묶음 |

| TMAP | 대중교통 경로를 제공하는 외부 서비스 | 출발 위치에서 관광지까지 이동 경로 |
| Kakao Local | 주소와 장소를 검색하는 외부 서비스 | 학교명이나 동네명을 검색해 위치 선택 |
| OAuth 로그인 | Google·Apple 계정으로 안전하게 로그인하는 방식 | 비밀번호를 Koready가 직접 받지 않음 |
| 테스트 서버 또는 staging | 개발 중인 기능을 팀이 확인하는 서버 | Render 환경 |
| 운영 서버 또는 production | 실제 사용자가 이용하는 정식 서버 | 향후 AWS EB 검토 |
| MVP | 가장 먼저 출시해 사용성을 검증할 최소 기능 범위 | 영상·오디오 가이드는 제외 |
| 상태값 | 여러 가능한 상태를 정해둔 값 | 축제 예정, 진행 중, 종료 |

## 3. 제품 정의와 MVP 범위

### 3.1 누구를 위한 서비스인가

Koready는 한국에서 비교적 오래 생활하는 외국인이 한국의 로컬 여행지를 발견하고, 실제 이동을 준비하며, 같은 장소에 관심 있는 사람과 연결될 수 있도록 돕는 서비스다.

- **핵심 사용자:** 외국인 유학생, 교환학생, 장기체류 외국인
- **확장 사용자:** 한국에 짧게 방문하는 단기 여행자
- **지원 언어:** 한국어와 영어
- **핵심 가치:** 관광지 발견, 취향 기반 추천, 현실적인 이동 안내, 여행 메이트 연결

단기 여행자를 막는다는 뜻은 아니다. 다만 초기 기능과 문구를 판단할 때 장기체류 외국인의 반복 사용을 우선한다.

### 3.2 MVP에 포함하는 기능

1. Google·Apple 로그인, 약관 동의, 언어 설정
2. 위치 입력 → 여행 스타일 1~4개 → 관심 관광지 1~3개의 온보딩
3. 홈과 월별 추천
4. 7개 권역 지도 탐색
5. K-Local Pick 추천 카드
6. 장소 검색, 상세 보기, 저장
7. Buddy Route와 짧은 Hori Tip
8. Buddy 공개 프로필, 장소별 메이트 조회, 비실시간 쪽지
9. 사용자 차단과 신고
10. 관리자가 관리하는 온보딩 후보 관광지 10개
11. 관리자가 작성하는 Hori Tip
12. KTO 데이터 수집 상태, 품질, 호출 기록과 관광공모전 증빙

### 3.3 MVP에서 제외하거나 뒤로 미루는 기능

- GPS를 이용한 기기 현재 위치
- 실시간 채팅, 상대방 입력 중 표시, 온라인 상태
- 여행 체크리스트
- 영상 가이드와 오디오 가이드
- 숙박시설 목록 추천
- KTX 가이드용 백엔드 API와 관리자 화면
- TMAP이 준 원본 경로의 장기 보관
- 사용자가 좌표를 직접 입력하는 기능
- KTO 원본 데이터를 가공 없이 프론트에 그대로 전달하는 기능

### 3.4 Hori 가이드 역할 분담

| 기능 | 담당 | 개발 방식 |
|---|---|---|
| Buddy Route 안의 짧은 이동 팁 | 백엔드 + 운영진 | 운영진이 작성해 저장하고, 경로 조회 때 조건에 맞는 팁을 함께 전달 |
| 코레일톡 화면을 따라가는 KTX 예매 가이드 | 프론트 | 기획에서 화면 순서와 문구·이미지를 전달하고 앱에 정적 콘텐츠로 제작 |
| KTX 예매 시뮬레이션 | 프론트 | 실제 예매 API 없이 안내 화면으로 제작 |
| 영상·오디오 가이드 | 제외 | MVP 이후 별도 검토 |

KTX 가이드를 위해 기획에서 프론트에 전달할 자료는 다음과 같다.

- 화면 순서와 각 단계의 제목
- 단계별 설명 문구와 주의사항
- 이전, 다음, 건너뛰기, 완료 버튼 동작
- 코레일톡 화면 이미지 예시와 잘라낼 범위
- 이미지 출처와 사용 권한
- 이미지가 보이지 않을 때 사용할 대체 설명
- 한국어·영어 문구
- 자료 버전과 관리 담당자

### 3.5 Figma에서 함께 고칠 내용

| 화면 | 현재 표시 | 개발 기준 |
|---|---|---|
| 위치 검색 | 현재 위치로 찾기 버튼 | 삭제하고 주소·학교·동네 검색만 사용 |
| 온보딩 | 방문 목적 단계 | 완전히 삭제 |
| 온보딩 순서 | 단계 번호가 중복됨 | 위치 01, 여행 스타일 02, 관심 관광지 03 |
| 경로 요약 | 편도 3시간 10분인데 당일치기 가능 | 편도 180분 이상이므로 숙박 권장 |
| 쪽지 | 최대 500자 | 최대 1,000자 |
| 지역 필터 | 강원 중복 | 7개 권역을 한 번씩 표시 |
| 월별 축제 | 연도와 종료 상태가 불명확 | 개최 연도, 기간, 예정·진행·종료 표시 |

    A[Google 또는 Apple 로그인] --> B[필수 약관 동의]
    B --> C[한국어 또는 영어 선택]
    C --> D[주소·학교·동네로 위치 입력]
    D --> E[여행 스타일 1~4개 선택]
    E --> F[후보 10개 중 관심 관광지 1~3개 선택]
    F --> G[홈]
    G --> H[월별 추천]
    G --> I[7개 권역 지도]
    G --> J[K-Local Pick]
    H --> K[장소 상세]
    I --> K
    J --> K
    K --> L[Buddy Route]
    K --> M[Buddy Connect]
    K --> N[장소 저장]
~~~

로그인은 개발 순서상 뒤로 미루지만 실제 제품에서는 첫 단계다. 로그인 개발 전에도 장소와 추천 기능을 테스트할 수 있도록 가짜 테스트 사용자로 검증한다. 실제 사용자 API를 인증 없이 공개하지는 않는다.

### 4.2 온보딩

1. 사용자가 Google 또는 Apple로 로그인한다.
2. 필수 약관에 동의하고 한국어 또는 영어를 고른다.
3. 주소, 학교명, 동네명을 검색해 체류 위치를 선택한다.
4. 여행 스타일을 중복 없이 1~4개 고른다.
5. 운영진이 미리 고른 관광지 10개 중 관심 장소 1~3개를 고른다.
6. 서버가 선택 개수와 후보 포함 여부를 확인한 뒤 온보딩을 완료한다.
7. 인터넷 문제로 완료 요청이 다시 전송돼도 선택 정보가 두 번 저장되지 않는다.

**방문 목적은 화면, API, DB 어디에도 만들지 않는다.**

위치 검색은 다음 방식으로 안전하게 처리한다.

- 사용자가 검색어를 Koready 서버로 보낸다.
- Koready가 Kakao에서 검색한 결과를 프론트에 전달한다.
- 사용자가 결과를 선택하면 약 10분 동안 유효한 확인표를 다시 서버에 보낸다.
- 서버는 확인표 안의 주소와 좌표만 저장한다.
- 프론트가 좌표나 권역을 임의로 바꿔 저장할 수 없다.

### 4.3 홈, 월별 추천과 축제

- 홈은 사용자의 기본 위치, 언어, 현재 연도와 월을 기준으로 필요한 내용을 한 번에 보여준다.
- 월별 목록은 조회할 **연도와 월**을 함께 받는다.
- 매년 반복되는 축제는 “같은 축제의 2026년 행사”와 “2027년 행사”를 별도 개최 건으로 관리한다.
- 축제는 시작일 6개월 전부터 해당 개최 연도·월의 상세 목록에 보일 수 있다.
- 축제가 끝나도 그 행사가 열렸던 연도·월의 목록에서는 바로 사라지지 않고 **종료**로 남는다.
- 홈과 현재 월 추천에서는 진행 중인 축제와 곧 열릴 축제를 종료된 축제보다 먼저 보여준다.
- 개최 날짜가 없거나 잘못된 축제는 날짜 기반 추천에서 제외한다.

예시:

> 2026년 10월 10일 시작하는 축제는 2026년 4월 10일부터 상세 목록에 노출할 수 있다. 10월 20일 종료 후에도 2026년 10월 목록에는 종료 상태로 남는다. 2027년 개최 정보가 들어오면 별도 행사로 저장한다.

### 4.4 지도와 장소 상세

- 전국을 서울, 경기·인천, 강원, 충청, 전라, 경상, 제주 7개 권역으로 나눈다.
- 지도 그림과 권역 버튼은 프론트가 담당한다.
- 사용자가 권역을 선택하면 백엔드가 해당 권역의 장소 목록을 전달한다.
- **내 주변**은 GPS 반경이 아니라 사용자가 저장한 기본 위치와 같은 7개 권역을 뜻한다.
- 장소 상세에서는 설명, 이동 경로, 여행 메이트 탭을 사용할 수 있는지 함께 알려준다.

### 4.5 K-Local Pick 추천

1. 사용자가 내 주변 또는 전국 추천을 연다.
2. 백엔드는 사용자의 여행 스타일과 장소 태그가 잘 맞는 후보를 먼저 고른다.
3. 추천이 한쪽에만 치우치지 않도록 일부 탐색용 후보를 섞는다.
4. 한 번 만들어진 추천 묶음에서는 장소와 순서가 바뀌지 않는다.
5. 같은 다음 페이지 요청을 다시 보내도 같은 카드가 나온다.
6. 카드가 API 응답에 포함된 시점부터 해당 장소는 30일 동안 새 추천 묶음에 나오지 않는다.
7. 후보가 부족하면 30일을 임의로 줄이지 않고 지역이나 매칭 범위를 넓힌다.
8. 그래도 후보가 없으면 더 이상 카드가 없다고 알려준다.

30일 정책은 MVP의 첫 기준이다. 실제 테스트에서 추천 후보가 너무 빨리 줄거나 빈 추천이 자주 나오면, 기록 시점이나 제한 기간을 새 정책 버전으로 조정한다. 이미 발생한 기록을 과거까지 바꾸지는 않는다.

개발 시에는 카드가 처음 응답에 포함된 시각, 당시 제한 일수 30일, 적용한 정책 버전을 함께 기록한다. 첫 정책의 시스템 이름은 recommendation-suppression-v1이다. 나중에 기준이 바뀌어도 어느 추천이 어떤 규칙으로 만들어졌는지 확인할 수 있다.

### 4.6 Buddy Route와 Hori Tip

1. 사용자가 저장한 출발 위치와 목적지를 선택한다.
2. Koready가 TMAP에 대중교통 경로 후보 최대 3개를 요청한다.
3. 실제 운행이 불가능하다고 표시된 구간이 있는 후보를 제외한다.
4. 총 이동시간, 환승 횟수, 도보 거리를 비교해 가장 적절한 경로를 고른다.
5. TMAP의 복잡한 원본을 프론트가 쓰기 쉬운 경로 요약과 구간 목록으로 바꾼다.
6. 편도 180분 미만은 당일치기 가능, 180분 이상은 숙박 권장으로 표시한다.
7. 운영진이 작성한 Hori Tip 중 목적지와 이동 구간 조건에 맞는 짧은 팁을 함께 붙인다.
8. 경로는 기본 약 30분 동안만 임시 보관하고, 만료되면 새로 검색하게 한다.

이동 난이도는 교통수단, 환승, 도보 거리를 합산해 쉬움·보통·어려움으로 표시한다. 초기 점수는 실제 TMAP 경로 표본을 더 확인한 뒤 조정하며, 어떤 계산 기준을 썼는지 버전을 남긴다.

#### 개발 참고: 이동 난이도 초기안

이 표는 확정 정책이 아니라 Route 개발 때 실제 표본과 비교할 출발점이다.

| 이동 요소 | 초기 점수 |
|---|---:|
| 지하철, 서울권 시내·마을버스, 비행기 | 구간마다 1점 |
| 지방권 시내·마을버스, 셔틀버스, 기차 | 구간마다 2점 |
| 시외·고속버스 | 구간마다 3점 |
| 환승 | 한 번마다 0.5점 또는 1점 후보 |
| 전체 도보 1,000m 이상 | 1점 추가 |

초기 구간 후보는 0~3점 쉬움, 3.5~6점 보통, 6점 초과 어려움이다. 환승 점수와 구간 기준은 아직 확정하지 않고 실제 경로를 확인한 뒤 정한다.

### 4.7 Buddy Connect

1. 사용자가 다른 사람에게 보여줄 Buddy 프로필을 만든다.
2. 장소 상세에서 같은 장소에 관심 있는 공개 프로필을 조회한다.
3. 상대에게 첫 쪽지를 보내면 해당 장소를 주제로 한 대화방이 생긴다.
4. 쪽지는 공백을 제외하고 1~1,000자다.
5. 인터넷 문제로 같은 전송이 반복돼도 쪽지는 한 건만 생성된다.
6. 자기 자신, 차단한 사용자, 수신을 거부한 사용자에게는 보낼 수 없다.
7. 실시간 채팅은 아니며 알림은 메시지 저장 성공 후 별도로 보낸다.
8. 차단은 즉시 목록과 메시지 가능 여부에 반영하고, 신고는 운영자가 별도로 처리한다.

### 4.8 관리자 운영

관리자는 별도 운영 화면 또는 도구에서 다음 일을 한다.

- KTO 데이터 수집 작업을 예약하거나 수동 실행한다.
- 일부 데이터 수집이 실패하면 실패한 항목만 다시 실행한다.
- 외부 API 호출 성공·실패와 데이터 품질을 확인한다.
- 온보딩 후보 10개를 만들고 검수한 뒤 공개한다.
- Hori Tip을 한국어·영어로 작성하고 노출 기간과 대상 장소를 정한다.
- 관광공모전 제출에 필요한 KTO 활용 증빙을 만든다.
- 관리자 권한과 중요한 변경 이력을 확인한다.

관리자 역할은 관리자, 운영자, 감사 담당으로 나누되 최초 관리자 계정을 만드는 절차는 개발 전에 한 번 더 정해야 한다.

## 5. 제품 운영 규칙

### 5.1 7개 서비스 권역

| 앱에 보이는 권역 | 포함 지역 |
|---|---|
| 서울 | 서울 |
| 경기 | 경기, 인천 |
| 강원 | 강원 |
| 충청 | 대전, 세종, 충북, 충남 |
| 전라 | 광주, 전북, 전남 |
| 경상 | 부산, 대구, 울산, 경북, 경남 |
| 제주 | 제주 |

이 권역은 추천과 지도 탐색을 단순하게 하기 위한 Koready 내부 분류다. 인천은 경기 권역에 포함한다.

### 5.2 여행 스타일

사용자는 다음 7개 중 1~4개를 고른다.

- 로컬 음식
- 지역 축제
- 전통시장
- 문화 체험
- 자연
- 전시·박물관
- 드라마 촬영지

KTO 데이터만으로는 드라마 촬영지 후보를 충분히 찾을 수 없다. 운영진이 먼저 장소를 선정하고, 검색어나 AI 분류는 후보를 찾는 보조 수단으로만 사용한다. 자동 분류만으로 공개하지 않는다.

### 5.3 온보딩 관광지 후보 10개

- 운영진이 장소, 순서, 대표 이미지, 한국어·영어 문구를 검수한 뒤 공개한다.
- 초안 단계에서는 10개보다 적을 수 있지만, 사용자에게 공개할 때는 정확히 10개여야 한다.
- 같은 장소나 같은 표시 순서가 두 번 들어갈 수 없다.
- 제목, 주소, 좌표, 대표 이미지가 있는 활성 장소만 공개할 수 있다.
- 공개된 10개 묶음은 직접 고치지 않고 새 버전을 만든다.
- 신규 사용자에게 보이는 현재 버전은 항상 하나다.
- 사용자가 이전 버전을 이미 본 경우, 새 버전이 공개된 뒤에도 이전 후보에서 고른 선택을 정상 제출할 수 있다.
- 지역이나 유형이 한쪽에 몰리면 관리자에게 경고하되, MVP에서는 자동 차단하지 않는다.

### 5.4 축제 상태

| 상태 | 사용자에게 보이는 의미 |
|---|---|
| 예정 | 아직 시작하지 않은 축제 |
| 진행 중 | 오늘이 시작일과 종료일 사이인 축제 |
| 종료 | 이미 끝났지만 해당 개최 연도·월 기록으로 남은 축제 |

상태는 매일 한국 시간 기준으로 계산한다. 별도 담당자가 상태를 직접 바꿀 필요는 없다.

### 5.5 Hori Tip

- 본문은 운영진이 직접 작성한다.
- TMAP이나 AI가 문구를 자동 생성하지 않는다.
- 공개 상태, 노출 기간, 목적지, 이동 구간 조건이 모두 맞을 때만 보인다.
- 경로 맨 위 또는 특정 이동 구간 다음에 붙는다.
- 같은 팁은 한 경로에서 한 번만 보여준다.
- 공개 전 한국어와 영어 문구가 모두 있어야 한다.
- 운영진이 비활성화하면 이미 조회한 경로에서도 다음 조회부터 팁이 사라진다.
- 범위는 Buddy Route의 짧은 이동 안내로 한정한다.
- 여러 팁이 동시에 맞으면 운영진이 지정한 우선순위가 높은 팁부터 보여준다.

## 6. 실제 외부 데이터로 구현 가능한가

### 6.1 프로파일링이란

문서에 적힌 API 설명만 믿지 않고 실제 데이터를 호출해 다음을 확인하는 조사다.

- 데이터가 실제로 몇 건 있는가.
- 장소 제목, 주소, 좌표, 이미지가 얼마나 채워져 있는가.
- 같은 ID가 중복되지 않는가.
- 문서와 실제 응답 모양이 다른 부분이 있는가.
- 오류가 어떤 형태로 오는가.

이 조사는 기능 완성이나 성능 보장을 뜻하지 않는다. **개발을 시작할 근거가 있는지 확인하는 단계**다.

### 6.2 KTO 조사 결과

2026-07-13에 실제 API를 호출했다. 전체 데이터 조사를 위한 호출 294회와 추가 확인 5회를 수행했고, API 키는 결과 파일에 저장하지 않았다.

| 데이터 | 확인한 건수 | 조사 범위 |
|---|---:|---|
| 한국어 관광정보 | 68,493건 | 전체 |
| 영어 관광정보 | 25,348건 | 전체 |
| 관광사진 | 6,119건 | 전체 |
| 관광공모전 수상작 | 96건 | 전체 |
| 연관 관광지 | 7,955건 | 여러 지역 표본, 전국 전체는 아님 |

한국어 관광정보 68,493건 중:

- 현재 노출 가능 표시가 있는 장소: 50,674건
- 좌표가 있는 장소: 50,622건
- 이미지와 주소가 있어 카드로 쓰기 좋은 장소: 42,517건
- 숙박과 여행코스를 빼고 온보딩 후보로 쓸 수 있는 기본 장소: 40,510건

주소와 좌표는 충분하지만 전화번호는 약 4.7%만 있어 필수 정보로 사용할 수 없다.

### 6.3 여행 스타일별 후보 수

| 여행 스타일 | 기본 후보 수 | 판단 |
|---|---:|---|
| 로컬 음식 | 9,831 | 충분 |
| 지역 축제 | 912 | 가능 |

따라서 드라마 촬영지를 제외한 여섯 스타일은 KTO를 기반으로 시작할 수 있다. 드라마 촬영지는 운영진의 초기 목록과 검수가 필요하다.

### 6.4 한국어 장소와 영어 장소 연결

KTO의 한국어 장소 ID와 영어 장소 ID는 서로 다르다. 숫자가 같다고 같은 장소로 보면 안 된다.

이미지 경로, 좌표, 관광 유형 등을 순서대로 비교한 결과:

- 영어 데이터 25,348건 중 20,106건, 약 79.3%를 한국어 장소와 자동 연결할 수 있었다.
- 여러 후보가 겹친 경우 1,142건은 자동 확정하지 않았다.
- 연결할 수 없는 데이터는 4,100건이었다.
- 이미지와 좌표 단서가 서로 충돌한 경우도 503건 있었다.

처리 원칙은 다음과 같다.

1. 한국어 장소를 기준 장소로 둔다.
2. 한국어와 영어 원본 ID는 별도로 보관한다.
3. 어떤 근거와 규칙으로 연결했는지 기록한다.
4. 후보가 여러 개거나 근거가 충돌하면 운영 검수 대상으로 보낸다.
5. 영어 원본이 없는 장소는 승인된 AI 번역 또는 수동 번역으로 보완한다.
6. 운영자가 직접 고친 번역은 자동 수집 작업이 덮어쓰지 않는다.

### 6.6 TMAP 조사 결과

실제 경로 요청은 3회 진행했다.

1. 서울 도심 정상 경로
2. 서울에서 김천으로 가는 장거리 정상 경로
3. 너무 가까운 거리를 요청했을 때의 오류

확인한 내용:

- 도보, 버스, 지하철, 기차 구간을 실제로 받았다.
- 공식 문서와 실제 응답의 영문 대소문자가 다른 필드가 있었다.
- 정상 경로 안에도 운행 불가 표시가 있는 구간이 포함될 수 있었다.
- 인터넷 응답 자체는 성공이어도 내용은 경로 검색 실패일 수 있었다.
- 이 차이를 고려한 변환과 오류 처리가 필요하다.

개발자는 TMAP의 “경로 없음” 계열 오류를 사용자가 다시 조건을 확인할 수 있는 오류로 바꾸고, TMAP 자체 장애는 잠시 뒤 다시 시도하도록 안내하는 오류로 바꾼다. 외부 서비스의 내부 오류 문구를 사용자에게 그대로 보여주지 않는다.

3회 시험은 기본 개발을 시작할 근거는 되지만 모든 지역과 교통수단을 검증한 것은 아니다. Route 개발 직전에 후보 3개 응답, 미래 출발시각, 영어 응답, 아직 보지 못한 교통수단을 소량 추가 확인한다.

## 7. 같은 데이터가 두 번 생기지 않게 하는 방법

### 7.1 현재 상태를 먼저 명확히 함

지금 Aiven DB에는 Koready의 장소·축제·추천 표가 아직 만들어지지 않았다. 따라서 아래 규칙은 **이미 작동 중인 기능이 아니라, 승인 후 첫 DB 작업에서 반드시 구현할 약속**이다.

KTO 조사 파일 안에서는 각 데이터 종류의 ID가 중복되지 않는 것을 확인했다. 실제 서비스에서는 단순히 “저장 전에 한번 검색해보기”만으로 부족하다. 두 요청이 동시에 들어올 수 있으므로 DB 자체가 두 번째 중복 저장을 거절하게 만든다.

### 7.4 특히 중요한 세 가지 사례

#### 반복 축제

“서울빛축제”라는 축제 자체와 2026년 개최, 2027년 개최를 나눈다. 2026년 날짜가 수정돼도 2027년 기간을 덮어쓰지 않는다. 같은 해 여러 번 열리는 행사까지 구분할 수 있도록 연도 안의 회차 번호도 둔다.

#### 온보딩 후보 10개

운영자가 공개한 버전은 그대로 보존한다. 새 장소를 넣고 싶으면 새 버전을 공개한다. 사용자가 이전 버전을 보고 선택 중이라면 그 선택을 완료할 수 있어야 한다.

#### 쪽지

프론트가 전송할 때 임의의 요청 키를 함께 보낸다. 네트워크 문제로 같은 내용과 같은 키가 다시 오면 기존 쪽지를 돌려준다. 같은 키로 다른 내용을 보내면 잘못된 요청으로 거절한다.

### 7.5 사용자에게 공개하는 ID

DB 안에서 사용하는 순번을 추천 묶음, 경로, 대화방 같은 주요 기능의 공개 이름표로 그대로 쓰지 않는다. deck_, route_, thread_, evidence_처럼 종류를 알 수 있는 접두어와 추측하기 어려운 임의 값을 조합한다.

- 이름표가 우연히 겹치면 DB가 저장을 거절하고 서버가 새 값을 만든다.
- 사용자는 이름표만 보고 다른 사용자의 데이터 개수나 생성 순서를 알 수 없다.
- 다음 추천 위치를 나타내는 커서에는 추천 묶음, 다음 위치, 형식 버전, 만료 정보를 담아 서버가 서명한다.
- 커서가 바뀌었거나 다른 사용자의 추천에 사용되면 거절한다.

## 8. 어떤 데이터를 저장하고 어떻게 보호하는가

### 8.1 원본과 서비스용 데이터를 분리

KTO에서 받은 원본과 사용자가 보는 가공 결과를 분리한다.

~~~text
KTO 호출 기록
  → 변경하지 않는 원본 파일과 파일 지문
  → KTO 원본 장소
  → Koready 기준 장소
  → 한국어·영어 문구, 이미지, 축제, 태그
  → 추천 결과
~~~

이렇게 분리하면 다음을 확인할 수 있다.

- 어떤 KTO 원본에서 현재 장소 정보가 만들어졌는가.
- 자동 연결이나 AI 번역에 어떤 규칙과 버전을 사용했는가.
- KTO에서 노출 중단된 장소를 사용자 화면에서 즉시 숨겼는가.
- 운영자가 직접 고친 문구를 자동 작업이 덮어쓰지 않았는가.

### 8.2 보관 원칙

| 데이터 | 보관 원칙 |
|---|---|
| KTO 원본 | 공모전 증빙과 재처리를 위해 비공개 저장소에 압축 보관 가능 |
| TMAP 원본 경로 | 파일, DB, 공모전 증빙에 장기 보관하지 않음 |
| Koready용으로 정리한 경로 | 기본 약 30분, 최대 24시간 미만 임시 보관 |
| Kakao 원본 검색 결과와 검색어 | 장기 보관하지 않음 |
| Google·Apple 인증 토큰 | 로그에 남기지 않고 필요한 값도 원문 대신 안전한 변환값으로 저장 |
| 사용자 위치 | 서비스에 필요한 최소 정보만 저장하고 삭제 즉시 추천·경로에서 제외 |
| Hori Tip | 운영 이력과 변경 기록을 위해 보존하고 물리 삭제 기능은 두지 않음 |
| 관리자 작업 기록 | 나중에 확인할 수 있도록 추가 전용으로 보관하되 비밀값은 제외 |

삭제된 계정과 위치 정보를 언제 완전히 지울지는 개인정보처리방침과 함께 PM 또는 법무 정책으로 확정해야 한다.

### 8.3 비밀정보 보호

- 실제 DB 비밀번호, KTO·TMAP·Kakao 키, 로그인 비밀키, JWT 키는 저장소에 올리지 않는다.
- 로컬에서는 .env.local, Render에서는 비밀 환경변수로만 관리한다.
- 공개 예시 파일에는 실제 값 대신 입력 위치만 표시한다.
- 인증 헤더, 토큰, 개인 위치, 실제 사용자 데이터는 문서와 테스트 예시에 넣지 않는다.
- 외부 API 호출 기록에서는 키를 가린다.
- 모든 PR에서 비밀정보 탐지 도구인 Gitleaks로 실제 키가 포함됐는지 자동 검사한다.
- 공모전 증빙에도 TMAP 원본 경로와 개인 위치를 포함하지 않는다.

- 프론트의 실제 사용자 흐름과 연결한다.

로그인을 뒤로 미룬다고 보안을 해제하는 것은 아니다. 1~2단계에서는 테스트 사용자로 개발하고, 실제 프론트 사용자 연결 전에 로그인을 완성한다.

### 9.5 4단계: 영어와 데이터 품질 보강

## 10. AI와 개발자가 지킬 품질 기준

### 10.1 테스트 주도 개발을 쉬운 말로

모든 기능은 다음 순서로 개발한다.

1. **실패하는 테스트 작성:** 아직 기능이 없기 때문에 예상한 이유로 실패하는지 확인한다.
2. **최소 기능 구현:** 테스트를 통과할 만큼만 구현한다.
3. **코드 정리:** 동작을 유지한 채 이름, 중복, 구조를 정리한다.
4. **전체 검사:** 다른 기능을 망가뜨리지 않았는지 확인한다.
5. **PR 검토와 테스트 서버 배포:** 팀이 변경 내용을 확인한다.

테스트를 삭제하거나 검사 강도를 낮춰 억지로 통과시키지 않는다.

### 10.2 자동으로 확인할 항목

- 계산과 추천 규칙 단위 테스트
- 프론트와 약속한 요청·응답 모양 테스트
- 로그인과 권한 테스트
- 실제 MySQL을 이용한 DB 구조와 중복 방지 테스트
- 코드 구조 규칙 검사
- 전체 코드 테스트 비율 80% 이상
- Docker 실행 가능 여부
- 비밀정보 포함 여부

### 10.3 핵심 완료 기준

#### 장소와 KTO

- 같은 데이터를 다시 수집해도 장소 수가 늘지 않는다.
- KTO에서 노출 중단된 장소는 사용자에게 보이지 않는다.
- 알 수 없는 외부 분류값 하나 때문에 전체 수집이 실패하지 않는다.
- 현재 서비스 장소가 어떤 원본에서 왔는지 추적할 수 있다.

#### 온보딩

- 방문 목적 필드가 화면·API·DB에 없다.
- 위치 → 여행 스타일 → 관심 관광지 순서로 중단 지점을 복구한다.
- 후보가 9개 또는 11개인 묶음은 공개되지 않는다.
- 사용자는 스타일 1~4개, 장소 1~3개만 선택할 수 있다.
- 완료 요청을 반복해도 데이터가 늘어나지 않는다.

#### 추천

- 같은 추천 묶음에 장소가 중복되지 않는다.
- 같은 추천 페이지를 다시 요청하면 같은 결과가 나온다.
- 노출된 장소는 30일 동안 새 추천에서 제외된다.
- 당시 적용한 정책 버전과 제한 기간을 확인할 수 있다.

#### 월별 축제

- 조회할 연도와 월이 필수다.
- 같은 축제의 2026년과 2027년 개최 건이 별도로 저장된다.
- 시작 6개월 전부터 보인다.
- 종료 후에도 해당 연도·월에는 종료 상태로 남는다.
- 홈에서는 진행·예정 축제가 종료 축제보다 먼저 나온다.

#### 위치

- 만료되거나 변조된 Kakao 검색 확인표를 거절한다.
- 프론트가 주소와 좌표를 직접 만들어 저장할 수 없다.
- 같은 위치를 반복 저장해도 중복되지 않는다.
- 한 사용자의 기본 위치는 하나다.

#### Route와 Hori Tip

- 인터넷 응답이 성공이어도 TMAP 내용이 오류이면 성공 처리하지 않는다.
- 운행 불가 경로를 정상 경로로 보여주지 않는다.
- 179분 59초는 당일치기, 180분은 숙박 권장이다.
- 만료된 경로는 새 검색을 안내한다.
- Hori Tip은 운영진이 작성한 정보만 노출한다.
- KTX 가이드용 백엔드 API는 만들지 않는다.

#### Buddy Connect

- 쪽지 1,000자는 가능하고 1,001자는 거절한다.
- 같은 전송 재시도로 쪽지가 두 개 생기지 않는다.
- 두 사용자가 발신 순서를 바꿔도 같은 장소 대화방은 하나다.
- 차단하거나 수신 거부한 상대에게는 보낼 수 없다.

## 부록 B. 프론트와 백엔드가 함께 쓰는 주요 상태 이름

영문 이름은 시스템끼리 오해 없이 주고받기 위한 내부 약속이다. 사용자 화면에는 한국어 또는 영어 번역 문구로 표시한다.

| 제품 의미 | 시스템 이름 |
|---|---|
| 한국어, 영어 | KO, EN |
| 로컬 음식, 지역 축제, 전통시장, 문화 체험, 자연, 전시·박물관, 드라마 촬영지 | LOCAL_FOOD, LOCAL_FESTIVAL, TRADITIONAL_MARKET, CULTURE_EXPERIENCE, NATURE, EXHIBITION_MUSEUM, DRAMA_LOCATION |
| 위치, 여행 스타일, 관심 관광지, 완료 | LOCATION, TRAVEL_STYLES, PREFERENCE_PLACES, COMPLETED |
| 축제 예정, 진행 중, 종료 | UPCOMING, ONGOING, ENDED |
| 내 주변, 전국 | NEARBY, NATIONWIDE |
| 초안, 공개, 보관 | DRAFT, PUBLISHED, ARCHIVED |
| Hori Tip 활성, 비활성 | ACTIVE, INACTIVE |
| 도보, 버스, 지하철, 고속버스, 기차, 비행기, 배, 셔틀버스 | WALK, BUS, SUBWAY, EXPRESS_BUS, TRAIN, AIRPLANE, FERRY, SHUTTLE_BUS |
| 쉬움, 보통, 어려움 | EASY, NORMAL, HARD |
| 당일치기 가능, 숙박 권장 | DAY_TRIP_AVAILABLE, STAY_RECOMMENDED |
| 한국어 초급, 중급, 고급 | BEGINNER, INTERMEDIATE, ADVANCED |
| 프로필 신고, 메시지 신고 | PROFILE, MESSAGE |

정확한 전체 상태값 35종은 openapi.yaml을 기준으로 자동 검사한다. 외부 서비스가 보내는 분류값은 예상하지 못한 새 값이 올 수 있으므로 그대로 보관하고, Koready가 프론트와 약속한 상태값으로 변환한다.
